# libhnefatafl — C Game Engine

Bitboard-based Hnefatafl engine with alpha-beta search, targeting x86-64 with BMI/BMI2/AVX2.

## Build

```
make              # shared lib + tests + benchmarks
make static       # static library (.lib/libhnefatafl.a) — needed by backend
make test         # build and run full test suite
make benchmarks   # build all benchmarks
make run-benchmarks
make clean
```

Debug build (assertions enabled, no `-DNDEBUG`): `make clean && make DEBUG=1`

## Compiler Flags

`gcc` with `-mbmi -mbmi2 -mlzcnt -mavx -mavx2 -O3 -Wall -Wextra -Werror -Winline`. All warnings are errors. Code must compile cleanly.

## Board Representation

11x11 board (121 squares, indexed 0-120). Throne at index 60, corners at 0/10/110/120.

```c
// 128-bit bitboard layer (two u64s for 121 squares)
typedef struct { u64 _[2]; } layer;

// Full board with rotated variants for algorithmic optimization
typedef struct {
  layer black, black_r;   // black pawns + 90° rotated
  layer white, white_r;   // white pawns + rotated
  layer king, king_r;     // king + rotated
} board;

// Compact representation for FFI/storage
typedef struct { layer black, white; u8 king; } compact_board;

// Simple move
typedef struct { u8 orig, dest; } move;
```

The `_r` (rotated) layers exist so move generation can use the same subtraction-difference ray algorithm for both axes. Every board mutation must update both the normal and rotated layers.

## Key Modules

| File | Purpose |
|---|---|
| `api.h/c` | Public FFI interface: `next_game_state()`, `search_trusted()`, `apply_move_sequence()` |
| `board.h/c` | Board struct, `apply_*_move()`, `to_compact()`/`from_compact()` |
| `layer.h` | 128-bit bitboard macros: `LAYER_OR`, `SET_INDEX`, `CHECK_INDEX`, `LAYER_POPCOUNT` etc. |
| `move.h/c` | Move generation via subtraction-difference rays. `all_black_moves()`, `all_white_moves()`, `all_king_moves()` |
| `move_generator.h/c` | Streaming move generator (state machine, `next_move()`) — avoids allocating full move lists |
| `capture.h/c` | Capture detection + application. Shield wall captures. Pre-computed `foe_masks[]`/`ally_masks[]` |
| `victory.h/c` | Win conditions: king capture, king escape, exit fort, surrounding, administrative endings |
| `king_mobility.h/c` | Exhaustive pre-computed king escape path analysis |
| `search.h/c` | Alpha-beta with iterative deepening, null move pruning, transposition table, PV tracking |
| `score.h/c` | Position evaluation: piece values, mobility, piece-square tables, corner guards |
| `zobrist.h/c` | Zobrist hashing with incremental updates |
| `transposition_table.h/c` | 4-entry buckets (64-byte cache line aligned), generation stamps |
| `validation.h/c` | Move and board state validation |
| `game_state.h/c` | Game state from move history, repetition detection via `position_set` |
| `io.h/c` | Board printing, algebraic notation, string formatting |
| `serialization.h/c` | Base64 encode/decode of move lists |

## Generated Constants

Several headers are generated by tools in `bin/`:

| Target | Generates | Content |
|---|---|---|
| `make gen_constants` | `src/constants.h` | Board region masks (throne, corners, edges, files, ranks) |
| `make gen_zobrist` | `src/zobrist_constants.h` | Random values for zobrist hashing |
| `make gen_masks` | `src/mask_constants.h` | Per-square `foe_masks[120]`, `ally_masks[120]`, `surround_masks[120]` |

Do not hand-edit these generated headers. Re-run the generator if the board representation changes.

## Testing

Framework: **greatest** (unit tests) + **theft** (property-based tests). Output colored via `greenest`.

```
make test                    # run full suite
make update-test-suites      # regenerate test.c from SUITE() macros in test files
```

Test files mirror source structure. Each declares `SUITE(name)` which is auto-discovered. To add a new test file: create `test/foo.c` with a `SUITE(foo)`, then run `make update-test-suites`.

## Benchmarks

Framework: **ubench.h**. Each benchmark file produces a separate binary.

```
make benchmarks        # build all
make run-benchmarks    # build + run all
make bench_capture     # build + run specific benchmark
make perf              # profile with Linux perf (search_perf target)
```

## Code Conventions

- **Naming**: `snake_case` for functions/types/variables, `UPPER_CASE` for macros
- **Layer macros**: `LAYER_*` prefix pattern (e.g. `LAYER_OR`, `LAYER_AND`, `LAYER_NOT`)
- **Formatting**: `clang-format` (run `./format.sh` from repo root). 80 columns, 2-space indent
- **Memory**: Caller frees dynamically allocated move arrays. Search uses stack allocation. TT allocated once
- **Inline**: Hot-path functions (move application, layer ops) are `static inline` in headers
- **Error handling**: Return codes and `move_error` enums for validation, not exceptions/longjmp

## Performance Notes

- Layer splitting (two u64s) is necessary for the 121-square board — don't try to use single integers
- The rotated board pattern is fundamental to move generation; removing it would require a full rewrite of `move.c`
- Transposition table entries are 16 bytes, packed 4 per 64-byte cache line
- The move generator (`move_generator.h/c`) streams moves one at a time for search — prefer it over `all_*_moves()` which allocates
